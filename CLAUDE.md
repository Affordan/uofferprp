# 强基信息系统项目总规则

你正在参与「强基信息系统」项目开发。
你的首要目标不是单纯写代码，而是稳定地交付一个可演示、可迭代、可追溯的数据产品。

## 1. 项目目标

本项目要构建一个展示中国强基计划信息的网站 demo，核心能力包括：

1. 展示各省强基计划政策与官方来源
2. 展示高校强基计划招生信息与官方来源
3. 查询和对比不同省份、不同高校、不同年份的强基计划分数线
4. 支持数据导出为 PNG 和 Excel
5. 具备后续扩展为正式系统的结构化数据基础

默认优先级：
数据正确性 > 来源可追溯性 > 可维护性 > 页面华丽程度

## 2. 工作方式

始终遵循以下流程：

1. 先理解需求与现有代码，再给出简短计划
2. 优先复用现有目录结构、组件、schema、工具函数
3. 在最小必要范围内实现，不做无关重构
4. 改动完成后，必须自行运行检查并修复问题
5. 输出结果时说明：
   - 改了什么
   - 为什么这样改
   - 还有哪些风险/假设
   - 下一步建议是什么

如果需求不完整，不要停在泛泛建议，先基于现有上下文做最合理实现，再明确列出假设。

## 3. 产品与数据原则

### 3.1 官方来源优先
涉及政策、简章、招生办法、分数线时：
- 优先使用教育主管部门、考试院、大学招生网等官方来源
- 尽量保留原始页面标题、发布时间、来源机构、原始链接
- 如果是 PDF 或公告页，优先保留原文件/原链接
- 不得把推测写成事实
- 遇到冲突来源时，必须显式标记冲突，不得私自合并结论

### 3.2 数据结构化优先
任何爬取、抽取、整理结果，都要尽量映射到稳定字段，而不是仅输出自然语言总结。

### 3.3 可比较优先
页面和接口设计要服务于“跨省份、跨高校、跨年份比较”，避免只适配单一页面展示。

### 3.4 可导出优先
涉及表格、图表、对比卡片时，要考虑后续 PNG / Excel 导出的稳定性。
避免难以截图、难以打印、难以导出的实现。

## 4. 推荐技术方向

默认按以下技术方向思考和实现，除非仓库已有不同方案：

- 前端：Next.js + TypeScript
- UI：Tailwind CSS + shadcn/ui
- 图表：ECharts
- 数据库：PostgreSQL
- ORM：Prisma
- 数据抓取与清洗：Node.js scripts 或 Python scripts
- 导出 Excel：xlsx 或等价方案
- 导出 PNG：基于页面区域截图或稳定的 DOM 转图片方案
- 测试：优先保证 lint、typecheck、基础单测、关键流程浏览器测试

如果仓库已有既定技术栈，以仓库现状为准，不要强推新栈。

## 5. 代码与目录约定

### 5.1 实现原则
- 优先写清晰代码，不炫技
- 优先小而稳的组件，不做过度抽象
- 优先补足类型，不放任 any 扩散
- 优先复用已有查询、schema、组件
- 优先把“来源信息”纳入数据结构

### 5.2 命名原则
- 字段命名尽量语义稳定、可复用、可搜索
- 年份、省份、高校名称、批次、分数类型等字段命名要统一
- 同一概念不要出现多个近义字段名

### 5.3 数据字段原则
涉及分数线时，优先拆分以下概念，不要混写：
- year
- province
- university
- major_or_track
- admission_stage
- score_type
- min_score
- avg_score
- min_rank
- notes
- source_title
- source_url
- source_date

## 6. 页面实现原则

目标风格：
- 简洁、现代、丝滑
- 信息密度高但不拥挤
- 强调搜索、筛选、比较、导出
- 空状态、加载状态、错误状态要清楚
- 对移动端和窄屏至少保持可用

页面体验原则：
- 数据列表先保证可读性，再考虑动画
- 比较页要突出差异，不要做花哨堆砌
- 详情页要突出来源、时间、规则、关键字段
- 导出区域要边界清晰，避免截图错位

## 7. 数据采集与清洗原则

当你编写爬虫、抽取器、清洗脚本时：
- 先分析来源页面结构，再设计字段映射
- 保留原始数据和清洗后数据，不直接覆盖
- 对异常值、缺失值、格式冲突要显式标记
- 不要静默吞掉错误
- 对无法确认的字段，输出“待人工确认”标记

## 8. 测试与验收原则

任何功能完成后，至少做以下检查：

1. lint
2. typecheck
3. 关键单测（如有测试体系）
4. 关键页面/关键接口的最小可用验证
5. 关键筛选、对比、导出链路是否可用

如果改动涉及：
- 表格渲染
- 筛选逻辑
- 数据清洗
- 导出能力
- schema 变更

则默认这是高风险改动，必须额外说明验证方式。

## 9. Git 与改动策略

- 一次任务只做一个明确目标
- 不要顺手大改无关文件
- 不要把格式化噪音混进功能提交
- 提交前先总结变更范围和风险
- 能做增量修改，就不要重写整块

## 10. Subagent 协作约定

如果项目中存在以下 subagents，优先委派对应任务：
- policy-collector：收集和核验政策/简章/公告官方来源
- score-normalizer：清洗和标准化分数线数据
- ui-implementor：实现或优化前端页面与组件
- test-reviewer：审查改动、复现问题、补测试、判断是否可合并

总原则：
- 主线程负责目标和决策
- subagent 负责专项任务
- 不把多个完全不同的问题塞给同一个 subagent

## 11. 输出格式要求

默认输出要简洁，但必须包含：
- Plan
- Changes made
- Validation
- Risks / assumptions
- Next step

如果是数据类任务，额外包含：
- Source summary
- Schema mapping
- Ambiguities

如果是页面类任务，额外包含：
- Affected routes/components
- Interaction states covered
- Export impact

## 12. 明确禁止

- 禁止伪造政策内容、分数线、来源链接、发布日期
- 禁止把猜测当正式数据写入结构化结果
- 禁止未经说明擅自删除旧字段或重命名核心字段
- 禁止只改展示层却忽略数据正确性
- 禁止在没有验证的情况下声称“已修复”
- 禁止为了通过测试而绕过真实逻辑
